signal()

Use a signal for writable reactive state. Read it by calling it like a function and update it with set or update.

const count = signal(0) count() count.set(1) count.update((value) => value + 1)

computed()

Use a computed signal for derived state. It tracks dependencies automatically and recomputes lazily when an input changes.

const price = signal(25) const quantity = signal(3) const total = computed(() => price() * quantity()) total()

linkedSignal()

Use a linked signal when the value should be derived from another signal but still stay writable between source changes.

const base = signal(100) const draft = linkedSignal(() => base()) draft.set(125) base.set(200) draft()

linkedSignal()

Draft is derived from base but stays writable. Changing base resets the draft.

    API Reference

    The core package stays small. Most helpers build directly on top of signal(), computed(), and effect().

    API What It Solves Use It When
    signal(initial, options?) Writable reactive state with set, update, mutate, asReadonly(), and optional custom equality. You own the source of truth and need explicit writes.
    computed(fn) Lazily derived state that tracks dependencies automatically. You need a value derived from other signals without storing duplicate state.
    effect(fn, options?) Reactive side effects with cleanup and optional rerun scheduling. You need to synchronize with logging, storage, DOM, timers, or network side effects.
    batch(fn) Coalesces multiple writes so dependents rerun once. You update several related signals in one logical transaction.
    untracked(fn) Reads signals without subscribing the current computation. You need the current value for branching, snapshots, or logging without dependency capture.
    linkedSignal(...) Writable derived state that resets or reconciles whenever its source changes. You need editable draft state that should still follow a parent signal.
    resource(...) Reactive async loading with cancellation, stale-value retention, and loading helpers. You fetch or recompute async data from another signal-driven request key.
    optimistic(source) Projects temporary local intent on top of settled state, then commits or rolls back later. You need optimistic UI for mutations without mutating the base signal early.

    When To Use optimistic()

    optimistic() is for mutations where the user should see intent immediately, but the actual source of truth still comes back from the server.

    • Reaction buttons Social feeds, likes, bookmarks, clap counters, and vote buttons should feel immediate even if the network takes 600ms.
    • Kanban and task reordering Move a card instantly in the UI, then commit the new order when the server accepts the mutation.
    • Chat and comments Insert a message or comment locally as “pending”, then finalize or remove it based on the server result.
    • Inline settings toggles Feature switches, follow/unfollow, star/unstar, and archive actions should reflect the click immediately while the mutation is in flight.

    Why Not Just Mutate Base State?

    • Rollback stays cheap A failed request just removes the optimistic layer instead of reconstructing previous state manually.
    • Server sync still wins If the backend pushes fresher data while a mutation is pending, optimistic layers rebase on top of the latest base signal.
    • Overlap is predictable Several optimistic actions can stack, settle, or fail independently without corrupting the source signal.
    • Cleaner mental model Base state is committed truth. Optimistic state is temporary local projection.

    optimistic() Example

    Typical mutation flow from real product code: show the new value immediately, then either commit it into base state or roll it back when the request settles.

    const serverLikes = signal(42) const optimisticLikes = optimistic(serverLikes) async function likePost() { const tx = optimisticLikes.apply((value) => value + 1) try { const result = await api.likePost() tx.commit(result.likes) } catch { tx.rollback() } }

    Use resource() for async reads. Use optimistic() for async writes that should feel instant.
